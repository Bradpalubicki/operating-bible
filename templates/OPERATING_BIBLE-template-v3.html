SLOs, Alerts & The Single Number
Define what "working" means in measurable terms. 1–3 SLOs per engine. Every alert links to an action. No alert without a runbook.
Why SLOs matter for NuStack: Without them, we optimize for the wrong thing — shipping features while the core loop quietly breaks. An SLO is just: "If THIS metric drops below THIS threshold, something is wrong and someone should know." That's it.
SLO table
Journey / Endpoint What we measure (SLI) Target (SLO) Window Alert name If broken →
[Core action — e.g., "Complete booking"] Success rate: POST /api/bookings 2xx / total ≥ 99% 7 days booking-success-rate Check DB, check Square creds, rollback if recent deploy
[Key page — e.g., "Dashboard load"] p95 response time: GET /dashboard p95 < 800ms 7 days dashboard-latency-p95 Check Supabase query performance, check N+1 queries
[Payment flow — e.g., "Payment collected"] Square webhook 200 acknowledgment rate ≥ 99.5% 30 days payment-webhook-ack Check Square dashboard, check idempotency key collisions
Golden signals dashboard (what to check first in any incident)

Latency

How long does it take to process a request? p50, p95, p99. Spike = query regression or cold start.

Traffic

How many requests/min? Sudden drop = upstream error or deployment issue. Sudden spike = check for abuse.

Errors

5xx rate. Any spike above 0.1% = investigate immediately. Link to Sentry project.

Saturation

DB connection pool %, Vercel function concurrency. Near limit = scale trigger.

Where to look
  • Vercel logs: vercel logs --environment production --since 1h --level error
  • Sentry: [Sentry project URL] — all unhandled exceptions, performance traces
  • Supabase: Dashboard → Database → Performance — slow queries, connection pool
  • PostHog: [PostHog project URL] — user-facing funnel drop-offs, event failures
Operations · v3.1
Incident Response
What to do when something breaks in production. Keep this short enough to actually use at 2am.
Severity taxonomy
SEV 0
Total outage — all users affected, core feature down
Examples: site down, auth broken, all payments failing. Action: rollback immediately, notify client within 15 min, fix forward. Do not sleep.
SEV 1
Core feature broken for some users, or data integrity risk
Examples: bookings saving but emails not sending, payment webhook failures >1%. Action: investigate root cause within 1h, notify client if user-visible, hotfix or rollback.
SEV 2
Degraded experience — not breaking, but wrong
Examples: slow page loads, UI bug visible to all users, non-critical feature broken. Action: fix in next deployment. Document in CHANGELOG.
SEV 3
Minor issue — cosmetic, affects <5% of users, no data risk
Examples: wrong copy, styling bug on one screen, edge case error. Action: add to backlog, fix in next sprint.
Standard playbook — SEV 0/1 response
1
Confirm impact
Is the user impact real? Check Sentry for error spike. Check Vercel dashboard — is latest deployment green? Check PostHog for funnel drop. If false alarm: document and close.
2
Stabilize first, diagnose second
If the bad deploy is <30min old: vercel rollback immediately. Don't spend time on root cause while users are down. Rollback takes 60 seconds.
3
Collect evidence
Pull Vercel logs (vercel logs --since 2h --level error --json). Screenshot Sentry issue. Note exact time incident started vs last deploy time.
4
Fix forward
Create fix branch. Write test that catches the regression. PR → merge → confirm deployment green → verify fix in production.
5
One-line postmortem
Add to CHANGELOG.md: [date] SEV[N]: [what broke] → [what was done] → [root cause] → [prevention]. That's all. No 10-page document for a NuStack incident.
Secret leak response (if API key appears in logs, repo, or Sentry)
Treat every suspected leak as a confirmed leak. Rotate first. Investigate after.
  • 1. Rotate the key immediately — Anthropic, Square, Supabase, Clerk, whoever issued it
  • 2. Add new key to Vercel env vars — vercel env add KEY_NAME production
  • 3. Redeploy — vercel deploy --prod or merge a trivial commit to main
  • 4. Check git log for the leak: git log --all -S "leaked_key_prefix" --oneline
  • 5. If key was in git history: rotate again after removing from history (or just keep it rotated)
  • 6. Add to .gitignore if the file that leaked it isn't already there

The Minimum Viable Complexity Law

"Does the client, on day one, need to see or interact with this to run their business?"

YES → build it simple. NO → hide it, defer it, or don't build it. Nothing from Layer 3 belongs on the dashboard. Ever.
